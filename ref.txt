Desenvolver uma aplicação web em **React** com backend em **SQLite** para controle de despesas e entradas de salário entre múltiplas pessoas. O sistema deve permitir **cadastro de pessoas**, escolha de cores a partir de **paletas pré-definidas**, entrada de dados via modal, gerenciamento de categorias, filtros, cálculos de saldos e visualização em cards e gráficos interativos.

### Estrutura dos Dados
Banco SQLite com quatro tabelas:

1. **Tabela `pessoas`**
   - id (PK)
   - nome (TEXT)
   - cor (TEXT) – escolhida dentro de uma paleta pré-definida

2. **Tabela `despesas`**
   - data (DATE)
   - valor (REAL)
   - origem_id (INTEGER → pessoas.id)
   - destino (TEXT: “Dividir” ou pessoas.id)
   - categoria_id (INTEGER → categorias.id)

3. **Tabela `salarios`**
   - data (DATE)
   - valor (REAL)
   - recebedor_id (INTEGER → pessoas.id)

4. **Tabela `categorias`**
   - id (PK)
   - nome (TEXT)

### Entrada de Dados
- **Botão “Adicionar pessoa”** → modal com Nome e escolha de cor (dropdown de paleta).
- **Botão “Adicionar despesa”** → modal com Data, Valor, Origem, Destino, Categoria.
- **Botão “Adicionar salário”** → modal com Data, Valor, Recebedor.
- **Botão “Adicionar categoria”** → modal com Nome da categoria.

### Paletas de cores pré-definidas
O sistema deve oferecer paletas temáticas, cada uma com 4–5 cores harmônicas.
- **Sunny Beach Day**
  - Charcoal Blue (#264653)
  - Verdigris (#2A9D8F)
  - Jasmine (#E9C46A)
  - Sandy Brown (#F4A261)
  - Burnt Peach (#E76F51)
- **Refreshing Summer Fun**
  - Sky Blue (#8ECAE6)
  - Blue Green (#219EBC)
  - Deep Space Blue (#023047)
  - Amber Flame (#FFB703)
  - Tiger Orange (#FB8500)

Cada pessoa cadastrada escolhe uma cor dentro da paleta.
Se houver mais pessoas do que cores em uma paleta, o sistema passa automaticamente para a próxima paleta.
A categoria **“Dividir”** mantém cor fixa vermelha.

### Regras de Negócio
- Ao abrir, todos os dados são exibidos.
- Filtros: Ano e Mês.
- Cálculos:
  - Total gasto por cada pessoa.
  - Total de despesas com destino “Dividir”.
  - Soma dos salários recebidos por cada pessoa no mês.
- **Card amarelo dinâmico:**
  - Sempre existe **um único card amarelo**.
  - Se houver duas pessoas → mostra quem deve pagar quem e o valor.
  - Se houver mais de duas pessoas → mostra lista de ajustes necessários (ex.: “Ana paga R$ 200 para João; Pedro paga R$ 150 para Ana”).

### Layout e Design
- **Cards:**
  - Card amarelo (saldo) no topo.
  - Cards abaixo: um para cada pessoa cadastrada (gastos).
  - Cards adicionais para salários de cada pessoa.
- **Gráficos:**
  - **Barra:**
    - X: Dia do mês.
    - Y: Valor (R$).
    - Mostrar todas as pessoas com suas cores escolhidas + “Dividir” (vermelho).
    - Bordas visíveis e sombra leve.
  - **Pizza:**
    - Proporção por categoria.
    - Sem legenda lateral.
    - Cada fatia com callout `"Categoria – XX%"`.
    - Fatias ordenadas em ordem decrescente.
    - Bordas visíveis e sombra leve.
- **Botões e filtros:**
  - Todos com sombra leve.
  - Layout responsivo (95% largura).
  - Tipografia amigável (Comic Sans ou similar).

### Comportamento esperado
1. Ao abrir, todos os dados são exibidos.
2. O usuário pode aplicar filtros de ano/mês.
3. O card amarelo mostra os ajustes de saldo entre as pessoas.
4. Cards abaixo mostram os totais de cada pessoa.
5. Cards adicionais mostram salários recebidos por cada pessoa.
6. O gráfico de barras mostra gastos diários com cores das pessoas + “Dividir” em vermelho.
7. O gráfico de pizza mostra categorias com callouts e ordem decrescente.
8. Gráficos e botões têm bordas e sombra leve.
9. Modais permitem adicionar pessoas, despesas, salários e categorias.
10. Após salvar qualquer entrada, o dashboard é atualizado automaticamente.
